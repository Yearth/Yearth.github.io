前言

经过这一段时间的学习，我们对于 vue 2.6 有了比之前更深的认识：了解了 vue 初始化的大致流程，数据响应式是如何实现的，vue 是如何进行异步更新的，diff 算法的细节，以及模板编译的流程和组件化机制。

似乎对于每个机制，我们都有一定的认识和理解，但目前这些知识是相对零散，碎片的，举个简单的例子：

所以我们需要将之前零散的知识串联袭来，帮助我们更深的记忆和理解 vue 源码。

我们希望能看到一条相对比较明显的主线，那么直接看源码显然是不合适的，即便我们已经对 vue 源码有了一定的了解，这也会让我们陷入繁杂的代码海，很快就迷失方向。

这个时候，我们应该利用调试来帮助我们梳理流程。

  • 要研究 vue 异步更新策略：需要更新 data
  • 要研究 vue diff 算法：同样，需要更新 data
  • 要研究 vue 模板编译：不需要额外操作，new Vue 即可
  • 要研究 vue 组件化机制：需要有 Vue.component

initState(vm)

前面的 initInjections(vm) 和后面的 initProvide(vm) 这里可以不用太关心。我们把注意力放在 initState，因为这是实现响应式的地方。

这里依次处理了：

  • props
  • methods
  • data
  • computed
  • watch

在做数据响应式的流程中，首先用 proxy 做了 data 的代理到 _data 上，然后在 observer 完成了数据响应式的核心功能。

这里有一些需要理清楚的点：

  • new Observer(value) 的第一个参数是 opt.data，即我们为组件写的 data
  • 每一个 Object 类型的数据才会有一个对应的 new Observer(value)，普通的数据是没有的
  • 在 Observer 的构造函数中会 new Dep()，所以每一个 Object 类型的数据具有一个 dep
  • 在 Observer 的 walk 中会遍历 value 这个 Object，然后对其属性调用 defineReactive 进行响应式处理
  • 在 defineReactive 中会再次 new Dep()，所以每一个 Object 下的属性也都具有自己的 dep
  • 如果 value 存在嵌套的 Object，则会在 defineProperty 中递归，调用的函数是 observe
  • 实现数据响应式的核心方法为 Object.defineProperty
  • 在 get：reactiveGetter 中实现依赖收集
  • 在 set：reactiveSetter 中实现触发更新
  • 在依赖收集阶段会建立 Watcher 和 Dep 的双向关系
  • 由于在依赖收集阶段建立了 Watcher 和 Dep 的关系，所以在触发更新阶段，可以通过 dep.notify() 遍历 Dep 管理的 Watcher 数组，调用其中的 update 从而触发更新

挂载

这里调用 $mount 进行挂载，那这个 $mount 是从哪儿来的？

这里借助调试器就能帮我们找到答案：

我们知道这里对基础的 $mount 进行了针对 web 平台的扩展。

而这里的核心功能是生成 render 函数，调用方法为 compileToFunctions。

顺着这条线可以继续挖掘：

  • compileToFunctions 来自 createCompiler
  • createCompiler 又来自 createCompilerCreator
  • 在 createCompilerCreator 中做了三个操作：
    • parse：解析生成 ast 树
    • optimize：优化 ast 树，做静态节点标记
    • generate：生成渲染函数 code

至此，渲染函数 render 已成功生成并且挂到 $options 上，紧接着，调用 mountComponent，在这里进行了挂载操作：

  • 首先调用钩子 beforeMount
  • 核心方法为 updateComponent，其执行流程为：
    • 先调用 vm._render() 生成 vnode
    • 再调用 vm._update() 将 vnode 转化为真实 dom
  • 执行了 new Watcher()，所以之后可以完成响应式的更改视图
    • Watcher 的第二个参数为 updateComponent，其作用为更新视图

这里的重点分别是 _render 和 _update，我们分别来看看。

_render

在 mount 扩展完成之后，已经生成了对应的 render，并且挂到 $options 上，所以这里我们可以通过 console.log(app.$options.render) 将其打印出来：

由于 patch 相对重要且复杂，所以这里先补上脑图，然后再深入分析 patch，它会涉及到 diff 算法，异步更新等。

可以看到，这里是通过 _c 创建节点的，而前面初始化的时候，我们已经知道，在 initRender 的时候，会生成 _c，其核心方法为 _createElement，而在这里，分两种情况进行处理：

  • 自定义组件：createComponent
  • html 保留标签：new VNode()

那么深入下去则可以看到对于自定义组件的处理细节：

  • 异步组件：createAsyncPlaceholder
  • 处理双向绑定：transformModel
  • 处理数据：extractPropsFromVNodeData
  • 函数组件：createFunctionalComponent
  • 安装组件的管理钩子：installComponentHooks
  • 最后 new Vnode() 并且返回

而 _update 的核心功能都在 __patch__ 中，包括大名鼎鼎的 diff 算法，也是发生在这里的。

Patch

patch 中的处理大体分为两类：

  • 新节点为定义：删除
  • 老节点未定义：新增

其中第一个情况较为罕见，在新增的逻辑中，最核心的方法是 createElm，它的功能是创建一个新节点，并且这里创建节点要分两种情况处理：

  • 自定义组件：createComponent
  • html 保留标签：createChildren

其中在处理自定义组件的时候，会调用自定义组件的钩子函数完成初始化操作，进行 mount，至于自定义组件的注册，待会儿再讲。

然后在处理 html 保留节点的时候，实际上还是通过 createElm 递归创建的。

最后，在 patch 中，完成新节点的创建之后，会调用 removeVnodes 将老节点删除，这个时候就完成了界面的更新。

接下来应该补充的是，diff 算法的核心代码：patchVnode

patchVnode

diff 的逻辑比较复杂，大体可以分为已下几种情况：

  • 新老节点都有子节点，则调用 updateChildren
  • 新⽼节点都没有⼦节点，进行文本替换
  • 新节点有⼦节点⽽⽼节点没有⼦节点，则先清空老节点的文本内容，然后为其新增子节点
  • 新节点没有⼦节点⽽⽼节点有⼦节点，则移除老节点的所有⼦节点

上面的逻辑总结一下就是耳熟能详的：深度优先，同层比较。

然后在 updateChildren 中，vue 对比较算法进行了一些优化，具体表现在：

  • 新增了四个游标，分别指向新老节点首尾，这里分别用：osi，oei，nsi，nei 代指
  • 将新老节点的首尾都取出来，这里分别用：osv，oev，nsv，nev 代指
  • 对节点进行遍历，退出条件为新老节点游标不重合，遍历中分为五种情况来处理：
    1. sameVnode(osv, nsv)：patchVnode
    2. sameVnode(oev, nev)：patchVnode
    3. sameVnode(osv, nev)：patchVnode，移动节点
    4. sameVnode(oev, nsv)：patchVnode，移动节点
    5. 其他情况：将 nsv 拿到老节点中去遍历查找：
      • 找到了：patchVnode，移动节点
      • 没找到：createElm 新增节点
  • 当遍历结束之后，则分为三种情况处理：
    1. osi > oei：说明有新增节点，调用 addVnodes 进行批量新增
    2. nsi > nei：说明有删除节点，调用 removeVnodes 进行批量删除
    3. 其他情况：不做任何处理

接下来我们应该顺着视图更新的路线去看看异步更新的流程。

在 watcher 的 update 中，实现异步更新的是 queueWatcher，而这里需要注意的有以下几点：

  • 通过 nextTick 将更新任务放入为任务队列
  • flushSchedulerQueue 即更新任务的核心方法为 watcher.run()
  • 在 run 中，实现更新的方法为 this.get()
  • 而这个 get 是外部传入的第二个参数，即 updateComponent
  • 实现异步的优先级如下：promise > MutationObserver > setImmediate > setTimeout
